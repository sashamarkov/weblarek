Критические требования
Без соблюдения этих требований ваш проект не пройдёт проверку, эти пункты обязательно попросит исправить ревьюер.
Требования к документации
В файле README.md в корне проекта приведено краткое описание проекта, инструкция по его запуску и документация на проект.
Документация должна содержать:
описание структуры базового кода, компонентов и моделей данных, из которых состоит приложение, и назначения каждого элемента;
описание программного интерфейса компонентов и назначение каждой его части;
для каждого класса должны быть описаны: назначение класса, конструктор, поля и методы с указанием типов данных, которые хранятся или используются, как параметры или возвращаемые из методов значения;
описание типов данных, с которыми работает приложение;
описание архитектурного паттерна  MVP.
Если проект содержит UML-схему структуры классов, то она в формате JPEG или PNG.
Исходный код проекта соответствует приведённой документации. Если в код проекта вносятся изменения, они отражены в документации.
Требования к архитектуре проекта
Понятная структура классов. Каждый класс имеет одно предназначение и не перегружен функционально.
Базовый код отображения выделен, нет дублирования кода по унаследованным блокам.
Нет дублирования данных, хранимых в разных классах отвечающих за работу с данными.
Все взаимодействия с интерфейсом производятся с использованием классов, отвечающих за отображение.
Все DOM-элементы, с которыми взаимодействует класс, сохранены в поля класса; их повторный поиск не производится или закэширован.
Все DOM-элементы, с которыми взаимодействует класс являются дочерними основному элементу разметки, за который отвечает класс.
Взаимодействие между классами реализуется вне самих классов, например через события или через передачу экземпляров этих классов.
Повторяющиеся части кода выделены в утилиты в виде функций или вспомогательных классов.
Запросы к API не интегрированы напрямую в компоненты отображения или модели данных.
Класс для работы с сервером должен использовать функционал класса Api через композицию или наследование.
Стиль написания кода
Форматирование кода:

код проекта отформатирован единообразно,
нет нарушений иерархии отступов.
Объявление переменных:
для объявления переменных используется let и const, не используется объявление через var;
если переменная не перезаписывает своё значение, она объявлена через const;
нет «магических значений», все константные значения записаны в переменные.
Именование переменных:
имена переменных и функций написаны в стиле camelCase;
имена переменных — существительные, отражающие то, что в них хранится;
если в проекте есть несколько переменных с похожими данными, по их наименованиям понятно, что хранится в каждой переменной;
имена классов и типов данных написаны в стиле PascalCase;
имена массивов — существительные во множественном числе;
имена функций начинаются с глагола, отражающего то, что они делают;
для именования не используется транслит и неуместные сокращения.
Требования к типизации проекта
Код типов должен быть добавлен к стартовому набору в едином файле src/types/index.ts.
Код типизации не содержит ошибок линтера и TS.
Объявленные типы и интерфейсы соответствуют тем структурам и данным, которые описаны в документации.
Тип any не используется без необходимости.
Не используется ослабление контроля типов с помощью инлайнового отключения линтера или настроек в tsconfig.json.
Описания одинаковых типов не дублируются. При необходимости используется объединение типов или дженерики.
В классах отдельно объявлен интерфейс для экземпляра и отдельно — для конструктора, если планируется передавать в качестве параметра класс по имени (конструктор).
Функциональность, заявленная в задании, работает корректно
Внешний вид и расположение элементов страницы соответствует макету.
Главная страница:

содержит каталог товаров;
при нажатии на карточку товара открывается модальное окно с информацией о товаре;
при нажатии на иконку корзины открывается корзина;
на иконке корзины отображается счётчик — он показывает количество товаров в корзине.
Просмотр товара:

показана детальная информация о товаре;
присутствует кнопка «Купить», если товар не был выбран покупателем;
при нажатии на кнопку «Купить» товар добавляется в корзину (если не был добавлен в корзину ранее);
если товар находится в корзине, кнопка должна быть заменена на «Удалить из корзины»;
при нажатии на кнопку «Удалить из корзины» товар удаляется из корзины;
после нажатия кнопки модальное окно закрывается;
если у товара нет цены, кнопка в карточке должна быть заблокирована и иметь название «Недоступно».
Корзина:

отображается список товаров, которые выбрал покупатель;
для каждого товара указана цена и присутствует кнопка удаления. Если нажать на кнопку, товар удаляется из корзины;
отображается общая стоимость товаров в корзине;
присутствует кнопка оформления покупки;
если в корзине нет товаров, кнопка оформления должна быть деактивирована;
если товаров нет в корзине — вместо списка товаров выводится надпись «Корзина пуста»;
при нажатии на кнопку «Оформить» открывается модальное окно с формой оформления товара.
Оформление товара:

Первый шаг:

выбор способа оплаты;
ввод адреса доставки;
если адрес доставки не введён, появляется сообщение об ошибке;
если способ оплаты не выбран, появляется сообщение об ошибке;
кнопка «Далее» может быть активной, если: на форме нет ошибок, выбран способ оплаты и поле адрес доставки не пустое;
при нажатии «Далее» открывается вторая форма оформления покупки.
Второй шаг:

ввод почты и телефона покупателя;
если одно из полей не заполнено, появляется сообщение об ошибке;
кнопка «Оплатить» может быть активной, только если на форме нет ошибок и поля почты и телефона не пустые;
при нажатии на кнопку оплаты выполняется передача данных о заказе на сервер, появляется сообщение об успешной оплате, товары удаляются из корзины, данные покупателя очищаются.
Требования ко всем страницам:

Модальные окна закрываются:

по клику вне модального окна;
по клику на иконку «Закрыть» (крестик).
Модальные окна не должны скролиться.
Хорошие практики
В этом списке собраны приёмы, которые помогут сделать проект ещё лучше. На данном этапе их использовать не обязательно — ревьюер примет работу без них. Однако такие хорошие практики помогут вам сделать портфолио более профессиональным. Опытные разработчики обратят на них внимание.
Код организован логично. Например, сначала идёт объявление переменных, затем объявление функций и в конце вызвано отображение данных на странице.
Нет лишнего кода: неиспользуемых переменных, закомментированных фрагментов, избыточной логики.
Удалены все console.log, alert и debugger, которые использованы во время отладки.
Логично построена иерархия классов:

Выделены базовые повторяющиеся части.
Базовый код используется не напрямую, а служит основной для реализации предметной области с помощью наследования или композиции.
Базовые классы являются абстрактными.
Используется не больше трёх уровней наследования.
Выделенные классы могут применяться в рамках всего проекта, то есть их можно легко переиспользовать.
Схемы структур классов соответствуют нотации UML и не содержат ошибок с точки зрения следования этой нотации.
